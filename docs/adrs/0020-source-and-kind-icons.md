# ADR-0020 · Source-group icon overrides + PostKind lucide rendering + cartouche frame

**Status:** Accepted
**Date:** 2026-05-15
**Deciders:** Paul

## Context

The app has **three independent icon systems** today, each with its own
storage model and inconsistent rendering pattern:

1. **WhatsApp source chips** (`/network`, `/network/spread`) — render
   `gps_chat_labels.icon` (a single emoji from Grant's Supabase view)
   inside a coloured-dot pill.
2. **PostKind chips** (FAB intent picker, post composer) — render an
   inline-hardcoded lucide icon per kind (e.g. `<CheckSquare>` for
   `tick_or_cross`). The `PostKind.icon` column (emoji) exists in
   Prisma but is unused at the render surfaces.
3. **PostCard kind chip** (`KindChip` in `post-meta.tsx`) — renders a
   palette tint + text label only, no glyph at all.

This ADR addresses three connected asks:

- The "GPS Action Network" source chip should display the brand mark
  (a raster image), not a generic 🎯 emoji.
- The "GPS Network ✅ or ❌" source chip and the `tick_or_cross`
  PostKind should both render a **lucide tick+cross overlap pair**
  instead of an emoji or a single `CheckSquare`.
- Source-group icons should sit inside a visually distinct **cartouche
  frame** (coloured-border pill) so users can read "this represents a
  WhatsApp source group" at a glance, separate from PostKind icons,
  filter chips, or feature glyphs.

## Options considered

- **A. Extend Grant's `gps_chat_labels` view with an `image_url`
  column.** Single source of truth, no override layer. **Rejected** —
  Grant's pipe ownership; coordination cost; we'd be asking another
  team to host our brand assets across providers.
- **B. Inline-hardcode per-slug overrides in a code constant.** Quick.
  **Rejected** — not admin-editable; every new source needs a code
  deploy; doesn't generalise.
- **C. Local `SourceIconOverride` table joined on slug.** Selected. Slug
  is stable (auto-generated by Grant's INSERT trigger from the label;
  re-renames don't re-generate per Grant 2026-05-11 Round 3). Same
  pattern as the existing `shared/styles/source-palette.ts` colour
  override map — extended from colours to images/glyphs and
  promoted from code to DB.
- **D. Reuse Grant's `gps_chat_labels.icon` as a discriminated string
  (emoji | URL | lucide name).** Rejected — Grant's view is read-only
  by design (PAUL_INTEGRATION.md quirk #4); we shouldn't conflate
  Grant-owned metadata with our overrides.

For PostKind:

- **E. New `PostKind.lucideIcon` column + central `<PostKindGlyph>`
  renderer.** Selected. Additive (nullable); existing `icon` emoji
  stays as the fallback path. Replaces the per-component
  hardcoded-lucide arrays with a single DB-driven source.
- **F. Drop emoji `icon` column entirely.** Rejected — emoji are
  cheaper to ship (no SVG runtime), and rendering admins might
  prefer them for new kinds.

## Decision

### 1. `SourceIconOverride` model

```prisma
model SourceIconOverride {
  /// Joins to `gps_chat_labels.slug` (Grant's source). Slug-keyed,
  /// not chat_id-keyed, because slugs are friendlier in admin UI and
  /// auto-generated stably from labels via Grant's INSERT trigger.
  slug      String   @id

  /// 'image' | 'lucide'. Emoji has no override path — it's the
  /// fallback when no override exists.
  iconKind  String

  /// Set when iconKind='image'. Either a `/public`-relative path
  /// (v1: committed assets) or an absolute URL (v2: Vercel Blob).
  imageUrl  String?

  /// Set when iconKind='lucide'. Maps to `server/lib/lucide-icon-registry.ts`.
  /// Currently one registered name: 'tick-cross-pair'.
  lucideKey String?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

Slug is the PK (not a separate uuid `id`) because:

- One override max per source — no need for a surrogate key
- Direct joins from `gps_chat_labels` are cheaper
- Admin URL is friendlier: `/admin/sourceIconOverride/gps-action-network`

No `iconKind` enum (Prisma) because Postgres doesn't make this an FK-
worthy concept; the discriminator is just a runtime check in the
render layer. CHECK constraint at the SQL level is sufficient.

### 2. `PostKind.lucideIcon` column

```prisma
model PostKind {
  // ... existing fields ...
  /// Lucide name from `server/lib/lucide-icon-registry.ts`. When
  /// present, takes precedence over the `icon` emoji at render time.
  /// Null means render via emoji fallback.
  lucideIcon String?
}
```

Same registry as `SourceIconOverride.lucideKey` — single source for
the name → lucide component map.

### 3. Lucide icon registry

`server/lib/lucide-icon-registry.ts` (and a shared name list for type
safety). Single registry consumed by both `<SourceBadge>` and
`<PostKindGlyph>`. Starts with these entries:

- `tick-cross-pair` — composite overlap pair (custom JSX, not a single
  lucide icon). Renders green-filled `Check` circle + red-filled `X`
  circle, ~35% horizontal overlap.
- `check` — single `Check` in a green circle (for `signal: promote`
  on `SignalBadgeRow`).
- `x` — single `X` in a red circle (for `signal: remove`).

Adding a new entry requires updating both the registry map and the
type union; no DB change.

### 4. `<SourceBadge>` component (cartouche)

The cartouche is a **pill with a coloured 1.5px border** matching the
source's `color`. Three size variants:

- `chip` (default) — full picker chip: cartouche + label, 26px icon slot.
- `compact` — gallery tile overlay: cartouche only, no label, 18px slot.
- `micro` — spread-trace timeline row: cartouche only, 22px slot.

Inside the cartouche, the icon slot renders one of:

- `<img>` from `imageUrl` (cropped to circle, `object-fit: cover`)
- `<TickCrossPair>` or other lucide-registry glyph
- Emoji fallback from `gps_chat_labels.icon`

The cartouche **only applies to source-group icons** — never to
PostKind chips, type chips, or filter chips. That's how the visual
language signals "this represents a WhatsApp source group."

### 5. `<TickCrossPair>` component

Two overlapping circles: green-filled `Check` (left, behind) +
red-filled `X` (right, in front), ~35% horizontal overlap, white
2px halo on each circle to keep contrast against any background.
Reusable as a standalone glyph anywhere (PostKindGlyph, SourceBadge,
intent-picker hero).

### 6. `<PostKindGlyph>` component

Reads `postKind.lucideIcon` → renders from registry. Falls back to
`postKind.icon` emoji. Replaces the per-component hardcoded-lucide
arrays in `KindPickerSheet` and `PostForm` with a single DB-driven
renderer.

## Consequences

### Storage / cost

`SourceIconOverride`: ~200 bytes/row × <50 rows ever = trivial.
`PostKind.lucideIcon`: nullable string column, no row cost.

### Migration

- New table (no data migration).
- New nullable column on `PostKind` (no backfill needed; existing rows
  remain emoji-only until populated).
- Data migration seeds two `SourceIconOverride` rows:
  - `gps-action-network` → image at `/source-icons/gps-action-network.jpg`
  - `gps-network-yes-no` → lucide `tick-cross-pair`
- Data migration sets `PostKind.lucideIcon = 'tick-cross-pair'` on
  the `tick_or_cross` row.

### Image hosting (v1)

GPS Action Network's logo lives at
`public/source-icons/gps-action-network.jpg` (committed to repo).
Storage decision deferred:

- **v1: `/public` assets.** Zero infra, version-controlled. Limits us
  to admin-via-code-deploy for new uploads. Fine for the first 1–2
  sources.
- **v2: Vercel Blob.** Server action uploads → returns URL → stored
  in `SourceIconOverride.imageUrl` as absolute URL. Unlocks admin
  upload UI without a deploy. Land when a 3rd source needs an image.

Render layer reads `imageUrl` as opaque — works for both `/public`
relative paths and absolute Blob URLs.

### Forward compatibility

- New lucide registry entries: 1-line addition to the registry map +
  the name union; no migration.
- Per-PostKind icon updates: admin edits `PostKind.lucideIcon` via the
  existing `/admin/postKind` surface.
- Per-source icon updates: admin edits `SourceIconOverride` via
  `/admin/sourceIconOverride` (entity-metadata entry added in same PR).

### What's NOT in this ADR (parked)

- Vercel Blob upload server action + admin upload UI (v2).
- Source `Group` table (separate concept — internal groups, not
  WhatsApp sources).
- Generalised "icon system across the entire app" (parked until a 3rd
  surface beyond source + kind needs unification).
